## Composition

A comparison helps choose between technical paths. Reach for this surface when
the user's first question is "which of these should I pick, and why?" — comparing
runtimes, deployment models, plans, or implementation options — not when there is
a single recommendation to report (the [brief](../brief)) or a running set of
tasks to operate (the [workspace](../workspace)).

**Shared-criteria comparison.** Use rows or compact tiles with shared operational
criteria (control, setup effort, data boundary, model support, runtime, cost,
risk, speed). Keep the criteria visible and parallel across every option — the
same labels for every alternative, divided by the warm `--noir-hairline` so the
matrix reads as one quiet lattice — so the reader compares like against like.
Where a cell carries pass/fail or readiness state, let it ride a 6px
`--noir-dot-*` rather than a colored cell. This is a structured-criteria
[tile](../tiles) variant: where a plain tile grid presents parallel *options*, a
comparison presents parallel *criteria* with a verdict.

**A calm recommendation.** Mark the recommended path with the off-white
`--noir-spine` on its row, an off-white fill, or the primary button position from
the [control system](../controls), and name the accepted tradeoff in one calm
sentence. Never use a bright winner badge,
a colorful score system, a generic pros/cons card set, or a pricing-table layout
when the task is an operational comparison.
